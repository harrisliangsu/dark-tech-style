DESIGN PRINCIPLES
Four ideas behind the system
Every architectural decision in VectorDB Cloud traces back to one of these four principles.
Storage-compute separation
Indexes live in object storage. Query nodes are stateless and scale on demand. A 10x traffic spike rents 10x query nodes, not 10x storage.
Hybrid index by default
HNSW for recall, IVF for memory efficiency, picked per-collection by the planner. You don't tune knobs — the system does.
Postgres on the wire
Speak the Postgres protocol. Use pgvector syntax. Existing libraries, ORMs, and observability tooling work without changes.
Multi-tenant by design
Namespace isolation is a first-class primitive — not an afterthought. One control plane safely runs ten thousand customers.
SYSTEM LAYERS
A four-layer architecture
Each layer has a single responsibility and its own scaling axis.
Layer 1 Edge & Routing
router.go
Postgres protocol parser, tenant-aware connection pool, query rewriter.
limiter.go
Per-tenant rate limit, per-query budget, slow-query kill-switch.
auth.go
SCRAM-SHA-256, OIDC, IAM federation, key rotation.
Layer 2 Query Engine
planner/
Cost-based plan selection across HNSW / IVF / brute-force per collection.
executor/
Vectorized SIMD operators, GPU offload for >10k candidate sets.
reranker/
Cross-encoder reranking on FP8 weights, <200µs per candidate.
Layer 3 Index & Storage
index/hnsw
Mmap'd graph, lock-free upserts, incremental compaction.
index/ivf
PQ-compressed centroids, 32x memory savings vs raw.
storage/s3
Tiered: hot (NVMe) → warm (gp3) → cold (S3 IA), driven by access stats.
Layer 4 Control Plane
scheduler/
Bin-packs query nodes onto compute pool, reschedules on hotspot detection.
tenant-mgr/
Lifecycle, quota, billing meters, soft-delete & restore.
observability/
OTLP traces, RED metrics, slow-query log shipped to customer's bucket.
